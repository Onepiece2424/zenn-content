# はじめに

Terraformで **CloudFront + S3 + API Gateway + Lambda** の構成を構築していたところ、少しハマったポイントがありました。

API Gatewayへ直接アクセスすると正常にレスポンスが返るにもかかわらず、CloudFront経由でアクセスすると `403 Forbidden` が返ってきます。

今回はその原因と解決方法を紹介します。

# 構成

今回の構成は以下の通りです。

```
ブラウザ
    │
    ▼
CloudFront
   ├── S3（静的コンテンツ）
   └── API Gateway
            │
            ▼
         Lambda
```

CloudFrontでは、下記のようなパスで、CloudFront の Behaviorを設定しています。

* `/` → S3
* `/api/*` → API Gateway


# 発生した問題

状況としては、元々、API Gatewayへ直接アクセスすると正常にレスポンスが返っていました。

```
$ curl https://xxxxxxxx.execute-api.ap-northeast-1.amazonaws.com/api/hello

{
  "message": "Hello from Lambda!"
}
```

しかし、CloudFront経由ではリクエスを送ると、下記のようになりました。

```
$ curl https://xxxxxxxx.cloudfront.net/api/hello

{
  "message": "Forbidden"
}
```


つまり、API Gateway経由で、Lambdaにアクセスした時は正常に動作しているのに、CloudFront経由で、API Gateway＋Lambda にアクセスした時だけ失敗するという状況でした。


# 原因
今回の現象が起きた原因としては、**CloudFrontのオリジンリクエストポリシーで、AllViewer を使用していたため**です。

AllViewer について

https://docs.aws.amazon.com/ja_jp/AmazonCloudFront/latest/DeveloperGuide/using-managed-origin-request-policies.html#managed-origin-request-policy-all-viewer

curlコマンドを用いて、CloudFront経由で、API Gateway＋Lambda にリクエストを送信した時、CloudFrontではOrigin Request PolicyとしてAllViewerを利用していました（イメージは、下記のような感じです）。

```hcl
data "aws_cloudfront_origin_request_policy" "all_viewer" {
  name = "Managed-AllViewer"
}
```



`Managed-AllViewer` はブラウザから送られてきた Header、Cookie、Query String をそのままOriginへ転送します。

この中には **Hostヘッダー** （WebブラウザがWebサーバーへアクセスする際に送信する「宛先ドメイン名」の情報）も含まれています。

例えばブラウザからCloudFrontへアクセスすると、

```
Host: xxxxx.cloudfront.net
```

というHostヘッダーになります。

CloudFrontはこのHostヘッダーをそのままAPI Gatewayへ転送していました。

しかしAPI Gatewayが期待しているHostは

```
xxxxxxxx.execute-api.ap-northeast-1.amazonaws.com
```

です。

Hostヘッダーが一致しないため、API Gatewayはリクエストを拒否し、

```
403 Forbidden
```

を返していました。


# 解決方法

解決方法としては、Origin Request Policy を下記のように、AllViewerExceptHostHeader へ変更しました。

AllViewerExceptHostHeader について
https://docs.aws.amazon.com/ja_jp/AmazonCloudFront/latest/DeveloperGuide/using-managed-origin-request-policies.html#managed-origin-request-policy-all-viewer-except-host-header

具体的には、下記のような感じで変更しました。

```hcl
data "aws_cloudfront_origin_request_policy" "all_viewer_except_host_header" {
  name = "Managed-AllViewerExceptHostHeader"
}
```

さらにBehaviorも、`origin_request_policy_id = data.aws_cloudfront_origin_request_policy.all_viewer.id` のようなかたちになっていたので、下記のようなかたちに変更しました。

```hcl
# CloudFront Distributiion
resource "aws_cloudfront_distribution" "s3_distribution" {

 ・・・

 ordered_cache_behavior {

  ・・・
  
    origin_request_policy_id =
      data.aws_cloudfront_origin_request_policy.all_viewer_except_host_header.id
  }
}
```

上記のポリシーは、AllViewerExceptHostHeader を使用しているので、 **Hostヘッダーだけを転送しない** 設定になっています。

Hostヘッダーが送られない場合、CloudFrontはOriginに合わせて、今回はAPI Gatteway がオリジンとなるので、ビューワーリクエストには、下記を自動的に付与します。

```
Host: xxxxx.execute-api.ap-northeast-1.amazonaws.com
```

その結果、API Gatewayが期待するHostヘッダーになる → リクエストが正常に処理される → Lambdaが実行される ようになりました。


# 学んだこと

今回の原因はAPI GatewayやLambdaではなく、**CloudFrontがOriginへどのようなHTTPヘッダーを転送しているか** という設定でした。

CloudFrontのBehaviorやOrigin Request Policyはキャッシュだけではなく、Originへのリクエスト内容にも大きく影響します。

CloudFrontをAPI Gatewayの前段に配置する場合は、`Managed-AllViewerExceptHostHeader` を利用することで、Hostヘッダーによる403エラーを防ぐことができます。


# まとめ

CloudFront経由でAPI Gatewayへアクセスした際に403 Forbiddenが返る場合は、

* API Gateway単体では成功するか
* CloudFrontのOrigin Request Policyは何を使用しているか
* Hostヘッダーがそのまま転送されていないか

を確認すると、原因を切り分けやすくなります。

今回の経験を通して、CloudFrontのOrigin Request PolicyがHTTPヘッダーの転送内容を制御していること、そしてHostヘッダーがAPI Gatewayとの連携において重要な役割を持つことを学ぶことができました。
