## はじめに

Dockerイメージを手動でビルドし、Amazon ECRへプッシュして、その後Amazon ECSを更新する作業は、慣れていても意外と手間がかかります。

そこで今回は、GitHub Actionsを利用して以下の処理を自動化しました。

- ソースコードを取得
- AWSへOIDC認証
- Amazon ECRへログイン
- Dockerイメージをビルド
- Amazon ECRへPush
- Amazon ECS(Fargate)へデプロイ
- サービスが正常になるまで待機

完成したWorkflowはこちらです。

```yaml:.github/workflows/deploy.yml
name: Deploy to ECS

on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest

    permissions:
      id-token: write
      contents: read

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: ${{ secrets.AWS_REGION }}

      - name: Login to ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push image
        env:
          ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
          IMAGE_TAG: ${{ github.sha }}
        run: |
          docker build --platform linux/amd64 -t $ECR_REGISTRY/${{ secrets.ECR_REPOSITORY }}:$IMAGE_TAG ./app
          docker push $ECR_REGISTRY/${{ secrets.ECR_REPOSITORY }}:$IMAGE_TAG

          docker tag $ECR_REGISTRY/${{ secrets.ECR_REPOSITORY }}:$IMAGE_TAG \
            $ECR_REGISTRY/${{ secrets.ECR_REPOSITORY }}:latest
          docker push $ECR_REGISTRY/${{ secrets.ECR_REPOSITORY }}:latest

      - name: Deploy to ECS
        run: |
          aws ecs update-service \
            --cluster fargate-rails-api-cluster \
            --service fargate-rails-api-service \
            --force-new-deployment \
            --region ${{ secrets.AWS_REGION }}

      - name: Wait for service stability
        run: |
          aws ecs wait services-stable \
            --cluster ${{ secrets.ECS_CLUSTER }} \
            --services ${{ secrets.ECS_SERVICE }} \
            --region ${{ secrets.AWS_REGION }}
```

この後は、作成したワークフローについてどのような処理を行なっているのかを1つ1つ解説していきます。

# Workflow全体の流れ

まず、今回作成したワークフローは、GitHubへPushすると、以下の順番で処理が実行されます。

![](https://static.zenn.studio/user-upload/7d6dc1c0d85f-20260701.png)


```text
Push
 ↓
Checkout
 ↓
AWS認証(OIDC)
 ↓
ECRログイン
 ↓
Docker Build
 ↓
ECR Push
 ↓
ECSへデプロイ
 ↓
サービス安定化を待機
```

これにより、Pushするだけで最新アプリケーションがECSへ反映・デプロイされます。



# ① Checkout

```yaml:.github/workflows/deploy.yml
- uses: actions/checkout@v4
```

上記の `uses` キーワードは、ワークフロー内で事前に定義されたアクションや再利用可能なワークフローを呼び出すための設定になります。

また、`actions/checkout@v4` は、GitHub Actionsのワークフロー内で実行環境（ランナー）に自分のリポジトリのソースコードをダウンロード（チェックアウト）するための公式アクションです。

このステップを実行しないと、ランナー上に Docker Build する対象のソースコードが存在しないため、多くのワークフロー内で利用されます。

https://docs.github.com/ja/actions/get-started/understand-github-actions#actions

https://zenn.dev/farstep/books/learn-github-actions/viewer/use-of-actions

https://www.hanachiru-blog.com/entry/2022/09/08/120000


# ② AWS認証（OIDC）

```yaml:.github/workflows/deploy.yml
permissions:
  id-token: write
```

```yaml:.github/workflows/deploy.yml
uses: aws-actions/configure-aws-credentials@v4
```

次は、OIDCを用いたAWS認証についてです。
以前は、AWSアクセスキーをGitHub Secretsへ保存する方法が一般的でした。
しかし、現在ではOIDC（OpenID Connect）を利用する方法が推奨されています。
理由としては、以下の通りです。

https://docs.github.com/ja/actions/concepts/security/openid-connect

https://zenn.dev/kou_pg_0131/articles/gh-actions-oidc-aws

## OIDCを利用するメリット

- アクセスキーを保存しなくてよい
- 一時的な認証情報を取得できる
- キー漏洩リスクを減らせる
- IAMロールで細かく権限管理できる

GitHub Actionsでは、OIDCトークンを発行し、そのトークンを利用してAWS IAMロールを引き受けます。

まず、`permissions` についてですが、これは **GitHub Actions が自動的に発行する `GITHUB_TOKEN` に、どの権限を与えるかを設定する項目**です。
つまり、**「GitHub Actions が GitHub に対して何をしてよいか」を決める権限設定**になります。

https://docs.github.com/ja/actions/reference/workflows-and-actions/workflow-syntax#permissions

次に、`id-token: write` についてですが、これは**GitHub Actions が GitHub の OIDC プロバイダーから OIDC トークン（JWT）を取得できるようにするための権限**です。

https://docs.github.com/ja/actions/how-tos/secure-your-work/security-harden-deployments/oidc-in-cloud-providers#adding-permissions-settings

https://docs.github.com/ja/actions/reference/security/oidc#debugging-your-oidc-claims

GitHub Actions が AWS にログインする際は、次のような流れで認証が行われます。

```text
GitHub Actions
      │
      │ ① OIDCトークンを発行してください
      ▼
GitHub OIDC
      │
      │ ② JWT（OIDCトークン）を返す
      ▼
GitHub Actions
      │
      │ ③ OIDCトークンをAWS STSへ渡す
      ▼
AWS STS
      │
      │ ④ IAMロールを引き受け、一時的なAWS認証情報を発行する
      ▼
一時的なAWS認証情報
```

この流れのうち、`id-token: write` が必要になるのは **①～②** の部分です。GitHub Actions は、この権限が付与されていることで GitHub の OIDC プロバイダーへ OIDC トークンの発行を要求し、取得したトークンを AWS STS に渡して認証を行います。

また、GitHub Actions の OIDC に関する[ドキュメント](https://docs.github.com/ja/actions/concepts/security/openid-connect#oidc-%E3%81%A8%E7%B5%B1%E5%90%88%E3%81%99%E3%82%8B%E6%96%B9%E6%B3%95-github-actions)「OIDC と統合する方法 GitHub Actions」と照らし合わせると、この `id-token: write` の役割は、**「③ ワークフロー ジョブのステップまたはアクションは、GitHub の OIDC プロバイダーにトークンを要求できる」** に該当すると考えられます。
その後、AWS STS がトークンを検証し、IAM ロールを引き受けることで、一時的な AWS 認証情報が発行されます。

![](https://static.zenn.studio/user-upload/cb6ec1048dba-20260704.png)




# ③ Amazon ECRへログイン

```yaml:.github/workflows/deploy.yml
uses: aws-actions/amazon-ecr-login@v2
```

上記の設定で、Docker PushするためにAmazon ECRへログインします。
イメージとしては、`docker login`を自動実行しています。
実行理由としては、ECRはログイン認証が必要だからです。ログインしないとワークフロー内の `docker push`は失敗してしまいます。

https://docs.aws.amazon.com/ja_jp/AmazonECR/latest/userguide/registry_auth.html

ログイン後は、

```text
xxxxxxxx.dkr.ecr.ap-northeast-1.amazonaws.com
```

のようなレジストリURLが取得できます。これを後続のDocker Buildで利用しています。



# ④ DockerイメージをBuild

```bash:.github/workflows/deploy.yml
docker build \
  --platform linux/amd64 \
  -t repository:SHA \
  ./app
```

ポイントは `--platform linux/amd64` です。
Apple Silicon（M1/M2/M3）環境ではARM64イメージが作成されます。

一方、ECSでx86環境を利用している場合は起動できないため、明示的にamd64イメージを作成しています。

また、

```text:.github/workflows/deploy.yml
IMAGE_TAG=${{ github.sha }}
```

としているため、

```text
f8a30c8
```

のようなコミットSHAがタグになります。コミットSHAをタグにするメリットとしては以下の通りです。

**コミットSHAをタグにするメリット**

- どのコミットから作られたイメージか分かる
- ロールバックしやすい
- デプロイ履歴を追跡できる



# ⑤ latestタグも更新

```bash
docker tag
docker push
```

コミットSHAだけでは最新イメージが分かりにくいため、

```text
latest
```

タグも同時に更新しています。

結果として、

```text
repository:3f92c81
repository:latest
```

の2つのタグが保存されます。



# ⑥ ECSへデプロイ

```bash:.github/workflows/deploy.yml
aws ecs update-service \
  --force-new-deployment
```

ここが一番重要な部分です。update-serviceは、Amazon ECSで稼働しているサービスの設定変更や、新しいコンテナバージョンへの更新（デプロイ）を行うためのAPIおよびコマンドで、ECSサービスに新しいタスクを起動して下さいと指示を出しています。

ECSはイメージが更新されただけでは、タスクの再起動を行いません。

`--force-new-deployment`を指定することで、新しいタスク定義を登録しなくても、ECSは現在のタスクを順番に終了し、新しいタスクを起動します。

https://docs.aws.amazon.com/cli/latest/reference/ecs/update-service.html

今回の構成では常に`latest`タグを更新しているため、このオプションだけで最新イメージへ更新できます。



# ⑦ サービス安定化を待つ

```bash:.github/workflows/deploy.yml
aws ecs wait services-stable
```

このコマンドを実行しない場合、Workflowはデプロイ直後に終了してしまいます。

もし起動に失敗してもGitHub Actionsでは成功扱いになってしまう可能性があります。

`aws ecs wait services-stable`を利用すると、新しいタスクが起動したり、ヘルスチェックに成功したり、サービスが安定するまで待機します。

https://docs.aws.amazon.com/cli/latest/reference/ecs/wait/services-stable.html

失敗した場合はWorkflow全体も失敗になるため、問題をすぐに検知できます。



# GitHub Secrets

今回利用したSecretsは以下です。

| Secrets | 内容 |
|---------|------|
| `AWS_ROLE_ARN` | AssumeするIAMロール |
| `AWS_REGION` | AWSリージョン |
| `ECR_REPOSITORY` | ECRリポジトリ名 |
| `ECS_CLUSTER` | ECSクラスター名 |
| `ECS_SERVICE` | ECSサービス名 |

機密情報をWorkflowへ直接書かずに済むため、安全に管理できます。



# まとめ

今回構築したWorkflowでは、GitHubへPushするだけでECS(Fargate)へのデプロイが自動で実行されます。

自動化された処理は次のとおりです。

- GitHubからソースコード取得
- OIDCによる安全なAWS認証
- Dockerイメージ作成
- Amazon ECRへPush
- Amazon ECSへデプロイ
- サービス安定化まで待機

特にOIDC認証を採用することで、アクセスキーを管理する必要がなくなり、セキュリティ面も向上しました。

CI/CDを導入することで、デプロイ作業の手間を減らすだけでなく、人為的なミスも防げるため、小規模なプロジェクトでも導入する価値は十分にあると感じました。
