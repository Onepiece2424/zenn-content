# 背景
AWS で VPC 内の DNS を設計していると、「この名前解決はどこに問い合わせに行くのか？」が分かりにくいことがあります。

特に Route53 Resolver の DNS 解決優先順位は公式ドキュメントにまとまった形で書かれていないため、挙動を理解していないと DNS トラブルの原因になります。

この記事では、VPC 内での DNS 解決の流れと優先順位を整理します。

# 前提：VPC 内の DNS の入り口
VPC 内の EC2 などのリソースは、DNS クエリを次のリゾルバーに送信します。

169.254.169.253

これは Amazon Route53 Resolver のエンドポイントで、VPC 内の DNS 解決を行います。

EC2
 ↓
Route53 Resolver
 ↓
各 DNS へ問い合わせ

Resolver は問い合わせ内容を見て、どこへ問い合わせるかを判断します。

# DNS 解決の基本ルール
Resolver の最も重要なルールは　**最も具体的なドメイン (Most Specific Match) が優先される** です。

例

example.com
test.example.com
db.test.example.com

この場合、Resolver は、db.test.example.com　のルールを最優先で使用します。

つまり より長く一致するドメインが勝つというルールです。

# Route53 Resolver の DNS 解決優先順位
実際の挙動を整理すると、Resolver は次の順序で解決を試みます。

① Private Hosted Zone
② Resolver ルール（条件付き転送）
③ AWS 内部ドメイン（システムルール）
④ . ルール（インターネット DNS）

それぞれ見ていきます。

# ① Private Hosted Zone（最優先）
VPC に関連付けられている Private Hosted Zone が最優先です。

例

corp.internal

レコード

db.corp.internal → 10.0.1.10

DNS フロー

EC2
 ↓
Route53 Resolver
 ↓
Private Hosted Zone
 ↓
10.0.1.10

ポイント

- VPC に関連付けられているゾーンのみ
- インターネット DNS より優先される

# ② Resolver ルール（条件付き転送）
Resolver ルールは、特定ドメインを別 DNS へ転送する仕組みです。

例

onprem.local → オンプレ DNS

DNS フロー

EC2
 ↓
Route53 Resolver
 ↓
onprem.local
 ↓
オンプレ DNS

オンプレミスと AWS を接続するハイブリッド DNS でよく使われます。

# ③ AWS 内部ドメイン（システムルール）
Resolver は AWS 内部ドメインのために 　**自動定義ルール（system rule）**　を持っています。

代表例

ec2.internal
compute.internal
region.compute.internal

例

ip-10-0-1-5.ec2.internal

DNS フロー

EC2
 ↓
Resolver
 ↓
AWS 内部 DNS
 ↓
10.0.1.5

この仕組みにより、EC2 ホスト名、AWS 内部サービスの名前解決が可能になります。

# ④ 「.」ルール（インターネット DNS）
最後に該当するルールがなければ、「.」のルールが適用されます。

これは すべてのドメインを意味するフォールバックルールです。

Resolver は再帰リゾルバーとして動作し、インターネット DNS に問い合わせます。

Resolver
 ↓
Root DNS
 ↓
TLD DNS
 ↓
権威 DNS

例

google.com
amazon.com
example.org

# まとめ
VPC 内の DNS 解決の流れをまとめると次のようになります。

EC2
 ↓
Route53 Resolver
 ↓
① Private Hosted Zone
 ↓
② Resolver ルール
 ↓
③ AWS 内部ドメイン
 ↓
④ . ルール（インターネット DNS）

さらに重要なルールは、　**「最も具体的なドメインが優先される」**　という点です。

# おわりに
Route53 Resolver の挙動を理解していないと、次のようなトラブルが発生します。

- Private Hosted Zone が想定外に優先される
- オンプレ DNS に転送されない
- AWS 内部 DNS が解決できなくなる

DNS は一度トラブルが起きると調査が難しいため、Resolver の優先順位を理解しておくことが重要です。
