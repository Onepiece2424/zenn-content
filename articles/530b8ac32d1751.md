# はじめに
今回は、Terraform、S3、CloudFrontを用いた静的コンテンツを配信するためのアプリを作成しました。
静的コンテンツの配信において「速く・安く・安全に」を実現することは、現代のWebインフラにおける基本要件です。
本記事では、Terraform・Amazon S3・Amazon CloudFront を組み合わせた静的コンテンツ配信基盤を取り上げ、「なぜこの構成を選ぶのか」という設計思想に焦点を当てて解説します。

単なる構築手順ではなく、各サービスの責務の分離・セキュリティ設計・IaC（Infrastructure as Code）の思想まで踏み込んで説明します。
これからTerraformを用いて、S3とCloudFrontを用いてアプリケーションを作成しようと検討している方の参考になれば幸いです。

# アーキテクチャの解説
今回作成したアプリケーションのインフラのアーキテクチャは、下記になります。
![](https://static.zenn.studio/user-upload/9a7282f7cdfd-20260701.png)

**全体構成図**
```
ユーザー
  │
  ▼
[CloudFront Distribution]   ← エッジキャッシュ・HTTPS終端・アクセス制御
  │
  │ OAC (Origin Access Control)
  ▼
[S3 Bucket]                 ← 静的ファイルのストレージ（非公開）
```

### 各コンポーネントの責務
 
この構成は「**関心の分離（Separation of Concerns）**」の原則に基づいています。各サービスに明確な責務を持たせることで、変更容易性・セキュリティ・スケーラビリティを両立します。
 
#### S3：信頼性の高いオリジンストレージ
 
S3の責務は**ファイルの永続的な保管**に限定します。重要な設計判断は「**S3への直接アクセスを禁止する**」ことです。
 
S3バケットをパブリックに公開する設計は一見シンプルですが、次のリスクがあります。
 
- HTTPSが保証されない（S3の静的ウェブサイトホスティングはHTTP）
- CloudFrontを経由しないアクセスを制御できない
- コスト管理が困難（リクエスト単価がCloudFront経由より割高）
これらを避けるため、バケットはプライベートに保ち、CloudFrontからのみアクセスを許可します。
 
#### CloudFront：配信・セキュリティ・パフォーマンスのゲートウェイ
 
CloudFrontには複数の責務を集約しています。
 
| 責務 | 内容 |
|---|---|
| HTTPS終端 | ACM証明書によるTLS処理 |
| エッジキャッシュ | 世界450以上のエッジロケーションへのキャッシュ配信 |
| アクセス制御 | OACによるS3への認証済みアクセス |
| セキュリティヘッダー | レスポンスポリシーでHTTPSリダイレクト等を設定 |
 
#### OAC（Origin Access Control）：S3とCloudFrontの信頼関係
 
旧来のOAI（Origin Access Identity）に代わる最新の認証方式です。OACはSigV4署名を使用してCloudFrontがS3へアクセスする際の認証を行います。これにより、**S3バケットポリシーでCloudFrontからのアクセスのみを許可**という最小権限の原則を実現できます。
 
#### Terraform：インフラの宣言的管理
 
インフラをコードで管理する最大のメリットは**再現性**と**差分管理**です。
 
- 環境（dev/stg/prod）を変数で切り替え可能
- `terraform plan` で変更内容を事前確認
- Git管理によるインフラ変更履歴の追跡

## Terraform コードの解説

### ディレクトリ構成
今回作成したディレクトリやファイルの構成は下記になっております。

**ディレクトリ構成と設計思想**

```
$ tree
.
├── index.html  # 静的コンテンツ
├── README.md
└── terraform　# インフラリソース管理ディレクトリ
    ├── environments　# ルートモジュール
    │   └── dev # 開発環境（environmentsディレクトリ直下にprodといった本番環境用リソース作成予定）
    │       ├── main.tf # モジュールの呼び出し
    │       ├── terraform.tf # プロバイダーやリモードバックエンドを管理
    │       ├── terraform.tfvars
    │       └── variables.tf # envなどの変数を定義
    └── modules　# 子モジュール
        └── s3_cloudfront
            ├── main.tf # S3やCloudFrontを定義
            └── variables.tf
```

この構成で特に工夫したのは 「モジュールを使った環境分離」 と 「変数による差分の吸収」 の2点です。


### 工夫① モジュールを使った環境分離

environments/ と modules/ を分けた最大の理由は、「インフラの定義（何を作るか）」と「環境の設定（どの値で作るか）」を切り離すためです。

モジュールを使わず、環境ごとにTerraformファイルをコピーした場合、次の問題が起きます。

```
# ❌ モジュールなしの場合
terraform/
├── dev/
│   ├── main.tf   ← devのリソース定義
│   └── ...
└── prod/
    ├── main.tf   ← prodのリソース定義（devとほぼ同じコードが重複）
    └── ...
```

S3のバケットポリシーを修正したいとき、dev/main.tf と prod/main.tf の両方を変更しなければなりません。変更漏れはインフラの乖離につながります。

モジュールを導入すると、構造が変わります。

```
# ✅ モジュールありの場合
terraform/
├── environments/
│   ├── dev/main.tf   ← "modules/s3_cloudfront を dev の値で使う" とだけ書く
│   └── prod/main.tf  ← "modules/s3_cloudfront を prod の値で使う" とだけ書く
└── modules/
    └── s3_cloudfront/main.tf  ← リソース定義はここだけ
```

バケットポリシーを修正するとき、触るのは modules/s3_cloudfront/main.tf の1箇所だけです。全環境に自動的に反映されます。

### 工夫② 変数による環境差分の吸収

環境間で異なる値（バケット名・環境名など）は、コードに直書きせず変数で渡します。

```
# ファイルの役割分担
variables.tf   → 「どんな変数が必要か」の定義（型・バリデーション）
terraform.tfvars → 「その変数に何を入れるか」の値（dev固有の設定）
```

この2ファイルを分けることで、コードの構造を変えずに環境の値だけを差し替えられます。stg環境を追加するときは、environments/stg/ ディレクトリを作り、terraform.tfvars に stg の値を書くだけです。モジュールのコードには一切手を触れません。

### environments/dev/variables.tf と terraform.tfvars：変数の定義と値を分離する
 
```hcl
# variables.tf：どんな変数が必要かの「定義」
variable "env" {
  type = string
}
```
 
```hcl
# terraform.tfvars：その変数に「何を入れるか」の値
env = "dev"
```
 
この2ファイルを分けることに意味があります。`variables.tf` は「このモジュールはどんな入力を受け付けるか」というインターフェースの宣言であり、`terraform.tfvars` は「このdev環境では実際にどの値を使うか」という環境固有の設定です。
 
stg環境を追加したいときは `environments/stg/` ディレクトリを作り、`terraform.tfvars` に `env = "stg"` と書くだけです。モジュールのコードには一切手を触れません。
 
### environments/dev/main.tf：モジュールの呼び出し
 
```hcl
provider "aws" {
  region = "ap-northeast-1"
}
 
module "s3_cloudfront" {
  source = "../../modules/s3_cloudfront"
 
  env = var.env
}
```
 
`environments/dev/main.tf` はモジュールの呼び出しのみに徹しています。S3・CloudFront・OACの詳細はモジュール側に完全に隠蔽されており、**このファイルを読むだけで「dev環境でs3_cloudfrontモジュールを使っている」という意図が一目で伝わります**。
 
### modules/s3_cloudfront/main.tf：モジュールの実装
 
#### S3バケットとバケットポリシー
 
```hcl
resource "aws_s3_bucket" "main" {
  bucket = "frontend-static-content-bucket-${var.env}-rio-20260630"
 
  tags = {
    Name        = "Frontend bucket"
    Environment = var.env
  }
}
 
# aws_iam_policy_document でポリシーを構造的に記述
data "aws_iam_policy_document" "origin_bucket_policy" {
  statement {
    sid    = "AllowCloudFrontServicePrincipalReadWrite"
    effect = "Allow"
 
    principals {
      type        = "Service"
      identifiers = ["cloudfront.amazonaws.com"]
    }
 
    actions = [
      "s3:GetObject" # 読み取りのみ許可
    ]
 
    resources = [
      "${aws_s3_bucket.main.arn}/*",
    ]
 
    condition {
      test     = "StringEquals"
      variable = "AWS:SourceArn"
      values   = [aws_cloudfront_distribution.s3_distribution.arn]
    }
  }
}
 
resource "aws_s3_bucket_policy" "main" {
  bucket = aws_s3_bucket.main.bucket
  policy = data.aws_iam_policy_document.origin_bucket_policy.json
}
```
 
バケットポリシーの記述に `jsonencode()` ではなく `aws_iam_policy_document` データソースを使っているのがポイントです。JSON文字列をベタ書きするより**Terraformのリソース参照（`.arn` 等）を直接使えるため、タイポや参照ミスをコード補完で防げます**。
 
`condition` ブロックで `AWS:SourceArn` にCloudFrontディストリビューションのARNを指定しています。「CloudFrontサービスからのアクセス」ではなく「このディストリビューションからのアクセス」に限定することで、他アカウントのCloudFrontによる意図しないアクセスを防ぎます。
 
#### locals：モジュール内の定数を整理する
 
```hcl
locals {
  s3_origin_id = "myS3Origin"
  my_domain    = "mydomain.com"
}
```
 
`locals` ブロックは、モジュール内で複数箇所から参照する値を一箇所にまとめるための仕組みです。`s3_origin_id` はオリジン定義とキャッシュビヘイビアの両方で参照されており、ハードコードを繰り返すより変更に強い構造になっています。
 
#### OACとCloudFrontディストリビューション
 
```hcl
resource "aws_cloudfront_origin_access_control" "default" {
  name                              = "default-oac"
  origin_access_control_origin_type = "s3"
  signing_behavior                  = "always"   # 常にSigV4署名を付与
  signing_protocol                  = "sigv4"
}
 
resource "aws_cloudfront_distribution" "s3_distribution" {
  origin {
    domain_name              = aws_s3_bucket.main.bucket_regional_domain_name
    origin_access_control_id = aws_cloudfront_origin_access_control.default.id
    origin_id                = local.s3_origin_id
  }
 
  enabled             = true
  is_ipv6_enabled     = true
  default_root_object = "index.html"
```
 
`signing_behavior = "always"` は「リクエストに既に署名があっても必ず上書きする」という設定です。CloudFrontがS3へアクセスする際、常にSigV4署名を付与することを保証します。
 
#### キャッシュビヘイビアの多段設計
 
この構成の特徴として、コンテンツの種類に応じて3種類のキャッシュルールを使い分けています。
 
```hcl
  # ① デフォルト：通常コンテンツ（TTL 1時間）
  default_cache_behavior {
    allowed_methods        = ["DELETE", "GET", "HEAD", "OPTIONS", "PATCH", "POST", "PUT"]
    cached_methods         = ["GET", "HEAD"]
    target_origin_id       = local.s3_origin_id
    viewer_protocol_policy = "allow-all"
    default_ttl            = 3600   # 1時間
    max_ttl                = 86400  # 最大1日
 
    forwarded_values {
      query_string = false
      cookies { forward = "none" }
    }
  }
 
  # ② /content/immutable/* ：変更されない静的アセット（TTL 最大1年）
  ordered_cache_behavior {
    path_pattern           = "/content/immutable/*"
    allowed_methods        = ["GET", "HEAD", "OPTIONS"]
    cached_methods         = ["GET", "HEAD", "OPTIONS"]
    target_origin_id       = local.s3_origin_id
    viewer_protocol_policy = "redirect-to-https"
    compress               = true
    default_ttl            = 86400    # 1日
    max_ttl                = 31536000 # 最大1年
 
    forwarded_values {
      query_string = false
      headers      = ["Origin"]  # CORS対応のためOriginヘッダーを転送
      cookies { forward = "none" }
    }
  }
 
  # ③ /content/* ：通常コンテンツ（TTL 1時間・圧縮あり）
  ordered_cache_behavior {
    path_pattern           = "/content/*"
    allowed_methods        = ["GET", "HEAD", "OPTIONS"]
    cached_methods         = ["GET", "HEAD"]
    target_origin_id       = local.s3_origin_id
    viewer_protocol_policy = "redirect-to-https"
    compress               = true
    default_ttl            = 3600   # 1時間
    max_ttl                = 86400  # 最大1日
 
    forwarded_values {
      query_string = false
      cookies { forward = "none" }
    }
  }
```
 
キャッシュ戦略を3層に分けた意図は次のとおりです。
 
| パス | TTL | 意図 |
|---|---|---|
| `/content/immutable/*` | 最大1年 | ハッシュ付きファイル名など更新されないアセットは長期キャッシュでCDNコストを削減 |
| `/content/*` | 最大1日 | 更新頻度が中程度のコンテンツは短めのTTLでユーザーへの反映速度を確保 |
| デフォルト | 最大1日 | その他のリクエストは汎用設定で対応 |
 
`compress = true` は `/content/` 配下でのみ有効にしています。圧縮はCPUコストを伴うため、頻繁にアクセスされるコンテンツパスに絞って適用しています。
 
#### 地理的制限と配信コストの最適化
 
```hcl
  price_class = "PriceClass_200"
 
  restrictions {
    geo_restriction {
      restriction_type = "whitelist"
      locations        = ["JP"]
    }
  }
```
 
`price_class = "PriceClass_200"` は日本を含むアジア・北米・欧州のエッジロケーションを使用する設定です。全世界（`PriceClass_All`）と比べてコストを抑えつつ、主要リージョンをカバーできます。
 
`geo_restriction` でホワイトリストを `["JP"]` に絞ることで、日本以外からのアクセスを403で遮断しています。サービス対象が日本国内に限定される場合、意図しない海外アクセスによるコスト増やスクレイピングのリスクを低減できます。
 

 
## まとめ
 
本記事で解説した設計のポイントを整理します。
 
| 設計判断 | 理由 |
|---|---|
| environments/modules の2層構造 | 環境分離と実装の再利用を両立する |
| variables.tf と tfvars の分離 | 変数のインターフェース定義と値を分離し、環境追加をシンプルに保つ |
| S3をプライベートに保ちOACで制御 | 特定ディストリビューションからのアクセスのみに限定し最小権限を実現 |
| aws_iam_policy_document の使用 | ポリシーを構造的に記述しリソース参照の安全性を高める |
| locals でモジュール内定数を管理 | 値の重複を排除し変更に強い構造にする |
| キャッシュビヘイビアの多段設計 | コンテンツの更新頻度に応じたTTLで配信効率とコストを最適化 |
| geo_restriction で日本のみ許可 | 対象外リージョンからのアクセスによるコスト増とリスクを低減 |
 
Terraform × S3 × CloudFrontの構成は、静的コンテンツ配信の事実上の標準パターンです。重要なのは「動くものを作る」ことではなく、**「なぜその設計にするのか」を語れる構成にすること**です。今回採用した environments/modules の2層設計は、将来的にstg・prod環境を追加する際も、モジュールを呼び出すだけで同一構成を展開できる拡張性を担保しています。
 
次のステップとして、カスタムドメイン（Route 53 + ACM）の導入や、GitHub ActionsによるCI/CDパイプラインの構築も検討してみてください。
